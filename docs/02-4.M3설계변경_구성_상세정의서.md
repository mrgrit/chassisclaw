좋아. 바로 **M3 구현 TODO를 파일 단위 작업지시서 형태**로 정리한다.

지금 핵심은 이거다.

기존 `target` 개념에 너무 많은 의미를 몰아넣어서,

* 등록 대상
* 실제 작업 대상
* subagent 연결 endpoint
* 테스트용 가짜 노드

가 다 섞여버렸다.

그래서 M3는 아래처럼 **개념 분리**가 핵심이다.

* **asset**: ChassisClaw가 관리하는 자산
* **target**: 실제 작업 시 선택된 실행 대상
* **subagent**: asset에 붙는 실행 엔드포인트
* **skill/playbook/tool/experience**: 그 위에서 동작하는 오케스트레이션 계층

---

# 1. M3 목표

M3 완료 기준은 다음 4개다.

1. **asset 모델 도입**
2. **remote vm 2대 onboarding 구조 반영**
3. **onboarding skill이 target이 아니라 asset/subagent 기준으로 동작**
4. **실행 시점에 asset → target으로 변환되는 흐름 확보**

즉, 지금처럼 `data/state/targets/local-agent-1.json` 두 개로 끝내는 게 아니라,
실제론 먼저 **asset 등록**이 되고,
그 asset의 subagent가 살아 있으면
그때 작업 가능한 **target 후보**가 되는 구조로 바꿔야 한다.

### 3-10) 함수명 변경

기존:

* `_probe_target`

* `_get_asset`

* `_materialize_target_from_asset`

### 3-11) `execute_stub()` 수정

현재 문제:

* skill_inputs에서 바로 target_id를 꺼내 probe함
* 이러면 애초에 asset 개념이 못 들어감

바꿀 구조:

* `skill_inputs["node_a_asset_id"]`
* `skill_inputs["node_b_asset_id"]`

흐름:

1. asset 조회
2. asset에 연결된 subagent endpoint probe
3. probe 결과로 execution target 생성/갱신
4. install/decision 수행
5. summary 작성

즉,
현재:

* precheck target exists
* probe target

변경:

* precheck asset exists
* probe asset subagent
* materialize execution target
* decision
* install/report

---

### 3-12) precheck 수정

기존 job:

* `check_targets_exist`

변경:

* `check_assets_exist`

결과 예:

```json
{
  "ok": true,
  "missing_assets": [],
  "found_assets": ["asset-vm-143", "asset-vm-144"]
}
```

### 3-13) probe 결과 구조 변경

현재는:

* requested_target_id
* reported_agent_id
* matched

바꿔야 할 형태:

* requested_asset_id

### 3-14) `_build_onboarding_summary()` 수정

이 함수는 거의 다시 다듬어야 함.

현재 summary 문제:

* target 중심
* identity mismatch도 target 기준
* install plan artifact append 중복도 있었음

수정 방향:

* `targets` 대신 `assets`
* `execution_targets` 별도
* blockers를 asset 기준으로 작성

권장 summary 구조:

```python
summary = {
    "project_id": ...,
    "selected_skill": ...,
    "precheck_ok": ...,
    "assets": [...],
    "execution_targets": [...],
    "decision": ...,
    "install": ...,
    "artifacts": [],
    "overall_status": ...,
    "blockers": [],
    "next_actions": [],
    "action_plan_ready": False,
    "execution_ready": False,
    "execution_ok": None,
    "failure_summary": None,
    "bootstrap_assets": [],
    "successful_bootstrap_assets": [],
    "failed_bootstrap_assets": [],
}
```

### 3-15) identity mismatch 판정 로직 수정

현재:

* `requested_target_id != reported_agent_id`

이건 근본적으로 이상함.
target id와 agent id를 직접 비교하면 안 됨.

바꿔야 할 판정:

* asset에 기대한 endpoint/base_url과 실제 probe endpoint가 일치하는가
* 또는 asset metadata에 기대 agent_id가 있으면 그와 비교

즉, 이런 식:

```python
identity_mismatch = any(
    asset.get("subagent_identity", {}).get("matched") is False
    for asset in summary.get("assets", [])
)
```

## F. install stub / bootstrap 방향 수정

### 3-19) install script 생성부 수정

현재 install script는 placeholder subagent 설치를 가정하고 있음.
M3에서는 최소한 아래로 가야 함.

* asset의 address 기반으로 원격 설치를 준비
* 또는 실제 remote subagent 이미 설치된 경우 probe만 수행
* placeholder local-agent 흐름 제거

즉, `install_subagent`는
“이 asset에 대해 필요한 설치 액션 계획”을 만들고
실행은 해당 asset의 접속 방식에 따라 되도록 해야 함.

---

### 3-20) install script에서 하드코딩 제거

지워야 할 것:

* `nonexistent_command_zzz`
* local only assumption
* `http://127.0.0.1:55123/health` 고정
* `CHASSISCLAW_AGENT_ID=%H` 같은 임시값 의존

바꿔야 할 것:

* asset address 기반 health URL
* 예: `http://192.168.208.143:55123/health`
* 또는 reverse tunnel/proxy 구조면 그 규칙 반영

# 6. M3 완료 판정

아직 **완료 아님**.

왜냐면 지금은 여전히:

* 온보딩이 target 중심
* local-agent 잔재 존재
* identity mismatch가 target/agent id 비교 기반
* remote vm asset 개념 미도입

이 상태라서.

M3 완료라고 하려면 최소 아래가 되어야 한다.

* remote vm 2대가 `assets`로 등록됨
* subagent probe 결과가 asset에 귀속됨
* 실행 가능한 target이 asset에서 파생됨
* onboarding summary가 asset 기준으로 바뀜
* fake local-agent seed에 의존하지 않음
