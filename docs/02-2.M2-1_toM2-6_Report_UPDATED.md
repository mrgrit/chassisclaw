# M2-1 ~ M2-6 중간 보고 (최신 정리본)

## 1. 문서 목적

이 문서는 M2 중반까지 어떤 기능이 붙었는지 빠르게 복구하기 위한 요약본이다.

핵심은 아래다.

- approval 흐름 연결
- install stub plan 생성
- failure modeling 시작
- target identity mismatch 검출

---

## 2. 구현된 흐름

대표 skill `ops.onboard.two_node` 기준으로 아래가 가능해졌다.

1. precheck
2. node A / node B probe
3. decision 생성
4. approval 필요 시 resolve 전이
5. 승인 저장
6. install stub plan 생성
7. summary artifact 생성

---

## 3. 추가된 구조

### 3.1 install stub plan
- package manager
- python version
- needs_sudo
- install script 초안
- action plan

### 3.2 failure modeling
- `command_not_found`
- `permission_denied`
- `timeout`
- `network_unreachable`
- `package_manager_missing`
- `python_missing`
- `unknown_failure`

### 3.3 identity mismatch
- requested target과 실제 reported agent identity 불일치 감지
- blocker 생성
- resolve 또는 clarification로 전이

---

## 4. 의미

이 단계부터 ChassisClaw는 단순 stub runner가 아니라, **운영형 판단과 보고를 흉내내는 stub workflow**를 가지게 되었다.

즉,

- precheck
- probe
- approval
- install plan
- failure classification
- summary

를 하나의 skill 실행 흐름 안에 담을 수 있게 되었다.

---

## 5. 현재 관점에서의 한계

이 문서는 아직 `target 중심` 설계를 전제로 쓴 부분이 있다. 새 채팅에서는 다음처럼 보정해야 한다.

- identity mismatch는 사실 target 문제가 아니라 **asset-agent 정합성 문제**다.
- local-agent seed 구조는 테스트 편의용 잔재다.
- 이후 M3에서는 asset 기준으로 개념을 다시 정리한다.
