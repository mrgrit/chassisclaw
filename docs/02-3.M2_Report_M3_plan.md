# M2 완료 보고

## 1. M2에서 구현한 핵심

M2에서는 M1에서 만든 뼈대 위에 **대표 skill이 실제 운영 흐름처럼 움직이도록 연결**하는 작업을 했다. 핵심은 `ops.onboard.two_node`를 기준으로, 단순 skill 선택 수준을 넘어서 **precheck → probe → decision → approval → install stub → report**까지 이어지는 구조를 만든 것이다.

구현 완료 항목은 아래와 같다.

### skill 실행 고도화

* `run_stub`로 skill 입력 검증 및 project plan 주입
* `execute_stub`로 job 순차 실행
* `job_results` 저장
* `onboarding_summary` artifact 생성

### target precheck / probe

* target 존재 여부 확인
* node A / node B health / capabilities probe
* 결과를 job 단위로 저장

### decision / approval 흐름

* probe 결과 기반 decision 생성
* `already_present / installable_with_approval / manual_bootstrap_needed` 판단 구조 도입
* approval이 필요한 경우 `needs_approval / resolve` 전이
* `approve_stub`로 승인 저장
* 승인 후 다시 실행 가능

### install stub 확장

* `install_subagent` job 추가
* install plan artifact 구조 추가
* target별 install stub script 생성 구조 추가
* action plan 생성 구조 추가
* install 실행 결과 artifact 구조 추가

### failure modeling

* 실행 실패를 구조화:

  * `command_not_found`
  * `permission_denied`
  * `timeout`
  * `network_unreachable`
  * `package_manager_missing`
  * `python_missing`
  * `unknown_failure`
* failure type 기반 remediation suggestion 생성
* failure를 `needs_clarification / needs_approval / replan`으로 연결할 수 있는 구조 추가

### identity mismatch 검출

* requested target id와 reported agent id 비교
* mismatch 시 blocker 생성
* `needs_clarification / resolve` 전이

## 2. M2에서 실제 검증된 흐름

### 정상 케이스

* skill 실행
* precheck 통과
* probe 통과
* decision = ready
* install = skip
* summary artifact 생성
* `planned / report`

### approval 필요 케이스

테스트 모드로 검증 완료:

* decision = approval_needed
* `needs_approval / resolve`
* approve 저장
* install stub plan 생성
* summary가 `install_planned` 반영
* `planned / report`

### identity mismatch 케이스

실제 현재 테스트 환경에서 검증 완료:

* `local-agent-2` 요청
* 실제 응답 `agent_id=local-agent-1`
* summary에 `identity_mismatch:local-agent-2`
* `needs_clarification / resolve`

---

## 3. 현재 상태를 한 줄로 정리하면

현재 csclaw는
**대표 skill 하나에 대해 사전검사, 프로브, 판단, 승인, 설치 준비, 보고까지 이어지는 운영형 stub workflow를 가진 상태**다.

즉 아직 완전한 자동 실행 시스템은 아니지만, 이제부터는 “구조를 새로 짜는 단계”가 아니라 **실행형 job과 planner 지능을 채워 넣는 단계**다.

---

## 4. 아직 stub인 부분

* install script는 bootstrap 초안 수준
* install action plan은 생성되지만 운영용 완성본은 아님
* 실제 remote transport/auth 다양화 미구현
* planner는 아직 rule/stub 성격이 강함
* failure remediation은 제안 수준이지 자동 실행까지 연결되진 않음
* run_auto와 skill runner가 완전히 하나의 상태기계로 합쳐지진 않음

---

# M3 목표

M3의 핵심은 이거다.

**“stub workflow를 실제 실행형 orchestration으로 끌어올리기”**

즉 M2까지가

* 선택
* 검사
* 판단
* 계획
  이었다면,

M3부터는

* 실제 실행
* 실행 결과 검증
* 실패 시 수정/재시도/재계획
  을 강화한다.

---

# M3 우선순위

## M3-1. install action 실제 실행 경로 고도화

* 현재 install script placeholder 줄이기
* bootstrap script를 더 현실화
* subagent 배포/실행/health 확인 연결 강화

## M3-2. failure remediation 연결

* failure type별로 실제 다음 액션 생성
* 예:

  * command not found → 도구 설치 승인 요청 또는 대체 도구 선택
  * permission denied → elevated execution approval
  * timeout → timeout 조정 후 재시도
  * network unreachable → connectivity probe branch
* 즉 분류를 넘어서 **실제 flow 분기**로 연결

## M3-3. run_auto 와 skill runner 통합

* 지금은 두 흐름이 병렬적임
* M3에서는 `plan → resolve → execute → validate → report` 공통 상태기계로 수렴
* skill 실행도 run_auto 체계 안으로 편입

## M3-4. target/transport 모델 강화

* target별 고유 endpoint
* ssh/http/local transport 분리
* auth 정보 모델 정리
* identity mismatch를 target integrity rule로 승격

## M3-5. evidence / artifact 체계 정리

* probe, install, execution, validation 결과를 일관된 artifact 체계로 정리
* 최종 보고 artifact 구조 표준화
