# M2 중간 완료 정리

## 1. 현재까지 구현된 범위

### M1에서 확보한 것

* `projects / targets / answers / approvals / run_auto` 기본 API
* project state 저장/조회
* `plan_ir` 기반 상태 관리
* `resolved_inputs → inputs / input_rationales / evidence_map` 반영
* `answer/approve` 후 재실행 시 state merge
* `validation / retry / replan` 최소 루프
* generic project 초기 상태 정리
* skill registry
* 대표 skill `ops.onboard.two_node`
* `run_stub` / `execute_stub` 기본 골격

### M2에서 추가된 것

* `decision → approval → install_subagent` 흐름 연결
* `approve_stub` 엔드포인트
* install job stub
* install plan artifact 구조
* install action plan 구조
* install execution 결과 구조
* failure classification 구조
* failure → `needs_clarification / needs_approval / replan` 연결
* target identity mismatch 검출 및 상태 전이

---

## 2. 현재 실제로 동작하는 흐름

지금 csclaw는 아래 흐름이 실제로 된다.

1. 프로젝트 생성
2. 대표 skill 선택 (`ops.onboard.two_node`)
3. 입력 검증
4. target existence precheck
5. node A / node B probe
6. capabilities / health 기반 decision
7. 필요 시 approval 상태 생성
8. approval 저장
9. install stub plan 생성
10. summary artifact 생성
11. failure / identity mismatch 시 resolve 또는 replan 전이

즉 이제는 단순 문서 구조가 아니라,
**skill 하나가 plan → probe → decision → approval → install stub → report**까지 도는 상태다.

---

## 3. 현재 확인된 중요한 동작

### 정상 케이스

* subagent가 이미 있는 target이면
* `decision = ready`
* `install = skip`
* `status = planned`
* `stage = report`

### approval 필요 케이스

* 테스트 모드에서 강제로 확인 완료
* `decision = approval_needed`
* `needs_approval / resolve`
* 승인 후 `planned / execute`
* 다시 실행 시 `approved_install_plan` 생성

### identity mismatch 케이스

* `local-agent-2`가 실제로는 `agent_id=local-agent-1` 응답
* `identity.matched = false`
* blocker 생성
* `needs_clarification / resolve` 전이

### failure classification 구조

* `command_not_found`
* `permission_denied`
* `timeout`
* `network_unreachable`
* `package_manager_missing`
* `python_missing`
* `unknown_failure`

이 분류를 바탕으로 나중에 질문/승인/재계획으로 보낼 준비가 됨.

---

## 4. 아직 stub인 부분

아직 아래는 진짜 실행형이 아니다.

* install script는 placeholder가 많음
* action plan은 생성되지만 실제 운영용 bootstrap 수준 아님
* decision 로직은 아직 rule 기반
* failure remediation도 아직 “제안” 수준
* LangGraph 상태기계로 완전 분해되진 않음
* experience / org profile / policy layer는 본격 미구현
* 원격 노드별 transport/auth 다양화 미구현
* 동일 subagent를 여러 target id로 보는 현재 테스트 구조 존재

---

## 5. 현재 코드 상태의 의미

지금 csclaw는
**“뼈대만 있는 문서 단계”는 이미 지났고, 대표 skill 하나를 끝까지 따라가 볼 수 있는 실행형 프로토타입”** 상태다.

중요한 점은,
앞으로는 매번 구조를 갈아엎기보다
이 위에 실제 job 실행과 planner 지능을 얹는 방향으로 가면 된다는 것이다.
