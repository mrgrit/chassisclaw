# M3 설계변경 - Asset 추가 (최신 실행 지시서)

## 0. 문서 목적

이 문서는 새 채팅에서 바로 구현을 재개할 수 있도록, Asset 추가와 관련된 **파일 단위 작업지시서**를 최신 기준으로 정리한 것이다.

핵심은 아래다.

- 기존 target 중심 구조를 해체한다.
- asset를 상위 관리 단위로 도입한다.
- 실행 시점에 asset을 target으로 resolve한다.
- onboarding skill을 asset 기준으로 바꾼다.

---

## 1. 목표

M3에서 Asset 추가의 목표는 다음 4개다.

1. **asset 모델 도입**
2. **원격 VM 2대 onboarding 구조 반영**
3. **onboarding skill이 target이 아니라 asset/subagent 기준으로 동작**
4. **실행 시점에 asset → target 변환 흐름 확보**

## 3. 파일 단위 작업 목록

## A. 모델 계층

### A-1. 새 파일 생성
`core/app/models/asset.py`

권장 필드:

- `id`
- `name`
- `kind` (`server`, `vm`, `container`, `device`)
- `address`
- `platform`
- `tags`
- `owner`
- `status`
- `subagent`
  - `installed`
  - `reachable`
  - `base_url`
  - `agent_id`
  - `last_seen`
- `metadata`

핵심:

- asset은 **관리 대상 그 자체**여야 한다.
- target 의미를 넣지 않는다.

### A-2. 기존 파일 수정
`core/app/models/target.py`

수정 방향:

- target에서 자산 의미 제거
- 실행용 뷰로 축소

권장 필드:

- `id`
- `asset_id`
- `resolver_type`
- `base_url`
- `mode`
- `capabilities`
- `tags`

추가 권장 함수:

- `build_target_from_asset(asset)`

### B-3. 기존 파일 수정

추가 또는 확장 필드:

- `selected_asset_ids`
- `asset_bindings`
- `resolved_assets`
- `execution_targets`

## D. 온보딩 로직 리팩토링

### D-1. 먼저 찾을 함수

```bash
grep -R "def execute_stub" -n core
grep -R "_build_onboarding_summary" -n core
grep -R "_probe_target" -n core
```

### D-2. 함수명 변경 권장

기존:
- `_get_target`
- `_probe_target`

변경:
- `_get_asset`
- `_probe_asset_subagent`
- `_materialize_target_from_asset`

### D-3. `execute_stub()` 수정

현재 문제:
- skill_inputs에서 target_id를 바로 꺼내 probe

변경 방향:
- `node_a_asset_id`
- `node_b_asset_id`

흐름:
1. asset 조회
2. asset에 연결된 subagent probe
3. probe 결과로 execution target 생성/갱신
4. decision
5. install/report

### D-4. precheck 수정

기존:
- `check_targets_exist`

변경:
- `check_assets_exist`

### D-5. probe 결과 구조 변경

기존:
- `requested_target_id`
- `reported_agent_id`
- `matched`

변경:
- `requested_asset_id`
- `expected_address`
- `reported_agent_id`
- `reported_base_url`
- `matched_asset_identity`
- `matched_endpoint`

### D-6. `_build_onboarding_summary()` 수정

summary 구조 권장:

```python
summary = {
    "project_id": ...,
    "selected_skill": ...,
    "precheck_ok": ...,
    "assets": [...],
    "execution_targets": [...],
    "decision": ...,
    "install": ...,
    "artifacts": [],
    "overall_status": ...,
    "blockers": [],
    "next_actions": [],
    "action_plan_ready": False,
    "execution_ready": False,
    "execution_ok": None,
    "failure_summary": None,
    "bootstrap_assets": [],
    "successful_bootstrap_assets": [],
    "failed_bootstrap_assets": [],
}
```

### D-7. identity mismatch 판정 수정

잘못된 비교:
- `requested_target_id != reported_agent_id`

바꿀 기준:
- asset에 기대한 endpoint와 실제 probe endpoint가 일치하는가
- asset metadata에 기대 agent_id가 있으면 그와 일치하는가

## 6. M3 완료 판정

아직 완료 아님.

완료로 보려면 최소 아래가 되어야 한다.

- remote VM 2대가 assets로 등록됨
- subagent probe 결과가 asset에 귀속됨
- 실행 target이 asset에서 파생됨
- onboarding summary가 asset 기준으로 바뀜
- fake local-agent seed에 의존하지 않음
