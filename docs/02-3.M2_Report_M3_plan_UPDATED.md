# M2 완료 보고 및 M3 계획 (최신 기준)

## 1. M2 완료 요약

M2에서는 M1의 코어 골격 위에 대표 skill이 실제 운영 흐름처럼 움직이도록 연결했다.

대표 skill `ops.onboard.two_node` 기준으로 다음이 가능해졌다.

- precheck
- probe
- decision
- approval
- install stub plan
- install execution stub
- report summary
- failure classification
- identity mismatch 검출

즉 M2까지 오면서 ChassisClaw는

> **“대표 skill 하나에 대해 사전검사, 판단, 승인, 설치 준비, 보고까지 이어지는 운영형 stub workflow”**

상태가 되었다.

---

## 2. M2에서 구현된 핵심

### skill 실행 고도화
- `run_stub`
- `execute_stub`
- `job_results`
- onboarding summary artifact

### target precheck / probe
- target 존재 여부 확인
- health / capabilities probe

### decision / approval
- `already_present`
- `installable_with_approval`
- `manual_bootstrap_needed`
- approval 저장 및 재실행

### install stub
- `install_subagent` job
- install plan artifact
- action plan
- install 실행 결과 구조

### failure modeling
- failure type 분류
- remediation suggestion
- resolve / approval / replan 연결 준비

### identity mismatch
- requested target id와 reported agent id 비교
- blocker 생성

## 3. 현재 상태 해석

M2의 의미는 “구조 설명 단계 종료”다.  
이제부터는 새 기능을 얹는 게 아니라, **실제 실행형 orchestration으로 진화시키는 단계**다.

다만 M2까지의 코드에는 중요한 구조적 한계가 남아 있다.

### 핵심 한계
- install script는 bootstrap 초안 수준
- install action plan은 운영용 완성본 아님
- planner는 아직 rule/stub 비중이 큼
- run_auto와 skill runner가 완전히 단일 상태기계로 통합되진 않음
- 가장 중요하게는 **target 중심 설계 잔재**가 남아 있음

---

## 4. M3의 재정의

초기에는 M3를 “install action 고도화” 정도로 보려 했지만, 실제 구현 과정에서 더 본질적인 문제가 드러났다.

### 드러난 문제
`target` 개념에 너무 많은 의미가 섞였다.

- 등록 대상
- 실제 작업 대상
- subagent endpoint
- 테스트용 local seed

이게 한 객체에 몰리면서,

- 같은 subagent를 서로 다른 target처럼 보거나
- identity mismatch를 잘못 해석하거나
- 온보딩 산출물이 target인지 asset인지 흐려졌다.

그래서 M3는 아래처럼 다시 정의한다.

> **M3 목표: asset 중심 구조를 도입하고, remote VM 2대를 실제 asset으로 온보딩하여, asset → target resolve 기반 실행 구조를 확립하는 것**

---

## 5. M3 핵심 과제

### M3-1. Remote SubAgent Bootstrap
- 원격 VM 2대에 실제 subagent 설치
- 서로 다른 `agent_id` 반환
- health / capabilities 정상화

### M3-2. Asset Registry 도입
- `data/state/assets/` 추가
- asset CRUD
- asset-agent 매핑 저장

### M3-3. Asset Identity Validation
- base_url uniqueness 검사
- asset 기대 identity와 실제 agent identity 비교
- mismatch를 asset registration 문제로 해석

### M3-4. Asset → Target Resolve
- target은 실행 시점 파생 객체로 축소
- asset 기반 선택 후 target materialization

### M3-5. `ops.onboard.two_node` 재구성
- 입력: asset 기준
- precheck: asset 존재 확인
- probe: asset에 연결된 subagent 검사
- install/report: asset 기준 요약

### M3-6. Summary / Artifact 정리
- asset 기준 summary
- execution target 분리
- identity mismatch 해석 수정

---

## 6. M3 완료조건

- 원격 VM 2대가 asset으로 등록됨
- 각 VM의 subagent가 서로 다른 identity를 반환함
- onboarding skill이 asset 기준으로 동작함
- 실행 시 asset이 target으로 resolve됨
- fake local-agent seed에 의존하지 않음

---

## 7. 새 채팅에서 바로 이어갈 실제 우선순위

1. 현재 코드/문서 checkpoint 확인
2. asset model 및 asset_store 추가
3. `data/state/assets/*.json` seed 또는 실제 등록 구조 추가
4. onboarding skill의 target 기반 로직을 asset 기반으로 리팩토링
5. remote VM 2대 실제 subagent 온보딩으로 테스트
