# M3 설계변경 상세정의서 (최신판)

## 0. 문서 목적

이 문서는 M3에서 왜 설계를 바꿨는지, 그리고 새 채팅에서 어떤 방향을 기준으로 코드를 손봐야 하는지 설명하는 **구조 변경 권위 문서**다.

이번 M3의 핵심은 단순 리팩토링이 아니라, **개념 모델 수정**이다.

---

## 1. 변경 배경

기존 구현에서는 `target`에 다음 의미가 동시에 들어가 있었다.

- 관리 대상 시스템
- 실행 대상
- subagent endpoint
- 테스트용 가짜 노드

이 때문에 아래 문제가 발생했다.

- 서로 다른 target이 같은 subagent를 가리키는 비정상 구조 발생
- `requested_target_id != reported_agent_id` 같은 어색한 identity mismatch 판정
- 온보딩의 결과가 target 등록인지 asset 등록인지 모호
- summary / blockers / next_actions가 실제 운영 개념과 어긋남

즉 문제의 핵심은 코드 버그만이 아니라 **모델 자체가 잘못 압축되어 있던 것**이다.

---

## 2. 새 기준 개념

## 2.1 Asset

관리 대상 자산의 기본 단위다.

예:

- 원격 VM
- 서버
- 네트워크 장비
- 관리 대상 서비스 인스턴스

Asset는 다음을 보관한다.

- 주소/접속 정보
- 플랫폼
- 태그
- 운영 상태
- 연결된 subagent 정보
- capability 정보

## 2.2 Agent / SubAgent

Asset 위에서 실제 실행을 수행하는 인터페이스다.

즉,

- Asset = 관리 대상
- Agent = 실행면

## 2.3 Target

특정 run에서 resolve된 실행 단위다. 고정 자산 DB 엔티티가 아니라, asset에서 파생되는 실행 컨텍스트 객체다.

---

## 3. 새 관계 구조

관계는 아래처럼 고정한다.

**사용자 요청 → Skill → Asset 탐색 → Target resolve → Playbook 선택 → Tool 실행 → Experience 축적**

즉,

- 등록은 asset
- 실행은 target
- 실체는 asset 위의 subagent

이다.

---

## 4. 구성요소별 책임 재정의

### Core
- 프로젝트 상태
- orchestration
- audit / evidence / artifact 권위 저장
- asset registry 조회
- target resolve

### SubAgent
- asset 위 실행면
- `/health`, `/capabilities`, `run_script`, file ops

### Bootstrap
- 원격 초기 설치
- subagent 배포
- health 확인
- asset 등록 전 상태 확보

### Engine
- 선택적 실행 엔진
- NanoClaw 등 외부 실행 계층 통합 지점

## 6. onboarding 흐름 재정의

기존 onboarding은 사실상 target onboarding처럼 동작하고 있었다.  
새 기준에서는 onboarding의 산출물을 아래처럼 본다.

### 자산 온보딩 단계
1. 원격 시스템 정보 입력
2. subagent 설치 또는 연결
3. `/health`, `/capabilities` 확인
4. asset registry 등록/갱신
5. asset-agent 매핑 저장

즉 산출물은 **target이 아니라 asset**이다.

### 작업 실행 단계
1. 사용자 요청 입력
2. skill이 asset 탐색
3. asset을 실행 가능한 target으로 resolve
4. playbook/tool 계획
5. approval/clarification 처리
6. 실행 및 검증

---

## 7. identity mismatch 해석 변경

이전에는 아래처럼 비교하는 잘못이 있었다.

- requested target id
- reported agent id

이제는 아래처럼 바꾼다.

- requested asset id
- expected endpoint/base_url
- expected agent identity(optional)
- reported agent id
- reported endpoint

즉 mismatch는 더 이상 `target mismatch`가 아니라 **asset-agent identity mismatch**다.

---

## 8. 현재 개발 단계에서의 실제 방향

### 실제 기준 자산
앞으로는 아래처럼 원격 VM 2대를 기준으로 본다.

- `vm-192-168-208-143`
- `vm-192-168-208-144`

### 실제 해야 할 일
- 각 VM에 subagent 설치
- 서로 다른 `agent_id` 부여
- Manager가 asset registry에 등록
- `ops.onboard.two_node`가 asset 기준으로 실행되도록 수정

---

## 9. M3의 실제 완료 기준

아래가 되어야 M3 완료로 본다.

- asset registry 존재
- remote VM 2대가 asset으로 등록됨
- 각 asset의 subagent가 서로 다른 identity 반환
- onboarding summary가 asset 기준으로 작성됨
- fake local target seed 없이도 테스트 가능

---

## 10. 새 채팅에서 이 문서를 어떻게 쓸 것인가

새 채팅에서는 이 문서를 **설계 권위 문서**로 사용하면 된다.  
즉,

- target 관련 코드를 읽을 때 이 문서를 기준으로 “무엇이 잘못된 잔재인지” 판단하고
- asset 관련 신규 모델/스토어/API를 설계할 때 이 문서를 기준으로 삼는다.
